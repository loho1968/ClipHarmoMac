## 二、追求目标可行性分析

### 追求目标 1：手机和Mac连接，实现剪贴板同步、发送图片和文件

**结论：✅ 已完全实现**

- 文字剪贴板同步：LAN TCP + 云中继 WebSocket 双模
- 图片发送/接收：两端均支持，含分片传输、压缩、写入系统剪贴板
- 文件发送/接收：两端均支持，含分片传输、保存到磁盘
- 手机端支持从相册选择图片、从文件管理器选择文件发送

### 追求目标 2：尽量无感自动连接和复制剪贴板（只需要文字），特别是公司、家里环境切换

**结论：✅ 已完全实现**

| 机制 | Mac 端 | 鸿蒙端 |
|------|--------|--------|
| LAN 自动发现 | UDP广播（3秒间隔） | 监听广播→自动TCP连接 |
| WiFi 变化重连 | `NetworkMonitor` 检测SSID变化→重启LAN服务 | `NetworkContextManager` 检测WiFi变化→匹配Profile→自动连接 |
| 双Mac切换 | 每台Mac独立roomKey，通过roomKeyInfo下发 | 匹配WiFi Profile后自动`switchRelayRoomKey` |
| TCP断线重连 | 持续UDP广播 | `TCPClient.scheduleReconnect()` 5秒间隔 |
| 中继断线重连 | `WSClient` 指数退避(1→30s) | `WSClient` 内建重连 |
| 后台保活 | LaunchAgent (KeepAlive=true) | `backgroundTaskManager` (MULTI_DEVICE_CONNECTION) |
| 熄屏恢复 | 响应 clipboardPoll | `startBackgroundPoll` 检测亮屏+每30s周期拉取 |

**公司→家里切换流程**：
```
1. 手机到公司，连公司WiFi → NetworkContextManager 匹配公司Mac Profile
2. 自动TCP连接公司Mac → 收到公司Mac的roomKeyInfo → 更新中继房间
3. 手机回家，连家里WiFi → NetworkContextManager 匹配家里Mac Profile
4. 自动TCP连接家里Mac → 收到家里Mac的roomKeyInfo → 切换到家里Mac的中继房间
5. 手机外出用5G → 无WiFi匹配 → 通过中继连接最近配对的Mac
```

### 追求目标 3：连接断了或者掉线，手机自动重连

**结论：✅ 已完全实现**

- TCP断线：手机端`TCPClient`自动5秒重连
- 中继断线：两端都有指数退避重连
- UDP发现：持续广播+去重重置（`discovery.resetFoundDevices()`），确保同一设备可被重新发现

### 追求目标 4：Mac重启后，自动连接

**结论：✅ 已完全实现**

- LaunchAgent (`RunAtLoad=true, KeepAlive=true`)
- 启动后自动开始UDP发现+TCP监听+中继连接
- 加密会话自动恢复（Keychain + UserDefaults持久化对端公钥）

## 三、剩余改进事项（按优先级排列）

### P1 — 体验优化

#### P1-1：Mac端 connectedDevice 显示设备名而非IP
- **现状**：`TCPServer.getRemoteAddress()` 返回的是IP地址
- **方案**：在UDP `ping` 消息中已有`deviceId`，TCP连接建立后从DiscoveryService的历史记录中映射IP→deviceId
- **文件**：`SyncManager.swift`
- **工作量**：约0.5天

#### P1-2：纯局域网用户减少中继连接尝试
- **现状**：`startRelayIfNeeded()` 无条件调用，若中继服务器不可达会产生后台错误
- **方案**：若配置文件不存在且从未手动设置过host，跳过连接。或增加静默失败日志级别
- **文件**：`SyncManager.swift`
- **工作量**：约0.5天

#### P1-3：多网卡广播从所有活跃接口发送
- **问题**：Mac有线+手机WiFi不同子网时，UDP广播可能走错接口。你在公司实际遇到了这个问题
- **方案**：枚举所有活跃网络接口（`getifaddrs`），对每个接口创建socket并绑定后发送广播（而非依赖单一socket的默认路由）
- **文件**：`DiscoveryService.swift`
- **工作量**：约1天

### P2 — 健壮性

#### P2-1：TCP连接健康检查
- **问题**：LAN侧无应用层心跳，TCP半开连接可能不会立即检测到
- **方案**：在TCP连接上增加应用层心跳（30秒无数据则发送ping）
- **工作量**：约0.5天

#### P2-2：剪贴板同步消息序列号
- **问题**：快速连续复制时，消息可能乱序到达
- **方案**：发送端加自增序列号，接收端按序处理
- **工作量**：约1天
