# 验证码自动复制 —— 鸿蒙 SMS 验证码自动同步至 Mac

> **状态**: 规划中  
> **优先级**: P3  
> **前置依赖**: 现有剪贴板同步通道（TCP/WebSocket 双模）  
> **预估工期**: 3 天  

---

## 1. 目标

鸿蒙手机收到验证码短信后，验证码自动同步到 Mac，Mac 弹出系统通知，用户一键粘贴。

- 手机端：剪贴板轮询自动发现验证码，通过现有通道发送
- Mac 端：收到验证码消息后弹出 macOS 本地通知、写入剪贴板、提供快速复制入口
- 无需用户任何额外操作（手机端仅需点击通知栏的"复制验证码"按钮，这是系统短信 App 自带功能）

## 2. 技术方案选型与结论

### 2.1 已评估方案

| 方案 | 平台 | 可行性 | 原因 |
|------|------|:---:|------|
| HMS Core ReadSmsManager | HarmonyOS 4.x (Android 双栈) | ❌ | Java/Android API，本项目是 HarmonyOS NEXT ArkTS 项目，架构不互通 |
| 华为帐号 SMS 自动填充 | HarmonyOS NEXT | ❌ | 仅限华为帐号登录场景，不能读取第三方短信验证码 |
| `ohos.permission.READ_MESSAGES` | HarmonyOS NEXT | ❌ | system_basic 权限，三方应用无法申请 |
| `ohos.permission.RECEIVE_SMS` | HarmonyOS NEXT | ❌ | system_basic 权限，三方应用无法申请 |
| `ohos.permission.NOTIFICATION_CONTROLLER` | HarmonyOS NEXT | ❌ | system_basic 权限，三方应用无法申请 |
| **剪贴板识别 + 同步** | HarmonyOS NEXT | ✅ | 利用现有剪贴板同步通道，增加验证码智能识别 |

### 2.2 平台决策依据

```
华为手机平台演进：
═══════════════════════════════════════════════════════════════

HarmonyOS 2.0/3.0/4.x                  HarmonyOS NEXT (5.0)
   (AOSP 兼容 + Android 双栈)              (纯鸿蒙微内核，去 AOSP)
         │                                       │
   ✅ HMS Core 可用                        ❌ HMS Core 不可用
   ✅ ReadSmsManager 可用                  ❌ 无 BroadcastReceiver
   ✅ Java/Kotlin 开发                     ✅ ArkTS 开发（本项目）
```

**结论**: ReadSmsManager 是面对 AOSP 兼容时代的 Android API。本项目基于 HarmonyOS NEXT ArkTS 技术栈，无法直接使用。剪贴板方案是当前架构下唯一可行且最优方案。

## 3. 用户操作路径

```
鸿蒙手机收到短信验证码
        │
        │ 用户在通知栏点击"复制验证码"（系统 Messages App 自带功能，1 次点击）
        ▼
验证码进入手机剪贴板
        │
        │ SyncManager.checkClipboard() 每 500ms 轮询（已有）
        │ 检测到 changeCount 变化
        ▼
readClipboard() → 正则识别是否为验证码（新增）
        │
        │ 发送 SyncMessage（type: verificationCode）
        │ 通过现有通道：LAN TCP 优先 → Cloud WebSocket 后备（已有）
        ▼
Mac 收到消息 → handleRemoteMessage()（新增验证码处理）
        │
        ├─→ macOS 系统通知横幅（新增）
        │   "验证码: 837261" + "复制验证码"按钮
        │
        ├─→ 写入 Mac 剪贴板（已有）
        │
        └─→ 状态栏菜单显示最新验证码（新增）
```

**用户操作次数**: 
- 手机端：1 次点击（点通知栏的"复制验证码"，这是系统自带功能，用户本来也会这样操作）
- Mac 端：1 次点击（通知横幅的"复制验证码" → 然后 Cmd+V 粘贴到输入框）

## 4. 验证码识别规则

### 4.1 识别正则（按置信度排序）

```typescript
// 需要从剪贴板内容中提取验证码的模式
const CODE_PATTERNS = [
  // 1. 中文上下文 + 6位数字（最常见）  "验证码 837261，请勿泄露"
  { regex: /验证码[^\d]*(\d{6})/,       confidence: 'high' },
  // 2. 中文上下文 + 4位数字            "您的验证码是 4928"
  { regex: /验证码[^\d]*(\d{4})/,       confidence: 'high' },
  // 3. 动态码/校验码 + 数字            "动态校验码 738291"
  { regex: /(?:动态|校验|安全)码[^\d]*(\d{6})/, confidence: 'high' },
  // 4. 英文 code + 数字                "code: 837261"
  { regex: /code[^\d]*(\d{4,8})/i,      confidence: 'medium' },
  // 5. 英文 OTP + 数字                 "OTP: 4928"
  { regex: /otp[^\d]*(\d{4,8})/i,       confidence: 'medium' },
  // 6. 纯独立6位数字行                 "837261"
  { regex: /^(\d{6})$/,                  confidence: 'low' },
  // 7. 纯独立4位数字行                 "4928"
  { regex: /^(\d{4})$/,                  confidence: 'low' },
]
```

### 4.2 识别策略

- **高置信度 (high)**: 同时匹配数字模式 + 中文上下文关键词 → 直接确认为验证码
- **中置信度 (medium)**: 匹配英文上下文 + 数字 → 确认为验证码
- **低置信度 (low)**: 仅有独立数字模式，无上下文 → 标记为候选，仍发送但降低通知优先级

### 4.3 去噪规则

- 忽略内容长度 > 200 字符（验证码短信通常很短）
- 忽略包含 URL 的内容（不是验证码短信）
- 忽略纯数字但包含常见非验证码模式（如电话号码 11 位、身份证号 18 位）
- 使用 `lastSentTimestamp` 去重（已有机制）

## 5. 协议扩展

### 5.1 Mac 端 Protocol.swift

```swift
enum MessageType: String, Codable {
    case clipboardText
    case clipboardImage
    case ping
    case pong
    case roomKeyInfo
    case verificationCode  // 🆕 验证码消息
}

struct SyncMessage: Codable {
    let type: MessageType
    let content: String
    let timestamp: Double
    let deviceId: String
    let mimeType: String?
    let networkSSID: String?
    var roomKey: String?
    var relayHost: String?

    // 🆕 验证码专用字段
    var codeValue: String?       // 提取出的纯验证码值，如 "837261"
    var codeConfidence: String?  // 置信度: "high", "medium", "low"
    var codeSender: String?      // 短信发送者号码（如有），如 "106900901234"

    // ... 其余不变
}
```

## 9. 边界与限制

| 场景 | 行为 |
|------|------|
| 剪贴板内容是普通文本（非验证码） | 走现有 `clipboardText` 通道，不做特殊处理 |
| 剪贴板识别为验证码但用户不想同步 | 鸿蒙端可增加"验证码自动同步"开关（默认开启） |
| 同一验证码重复触发 | `lastSentTimestamp` 去重 |
| 两条不同的验证码短时间内到达 | 各自触发独立通知，不会覆盖 |
| Mac 端未授权通知权限 | 仅写入剪贴板，不做通知；MainView 引导用户开启 |
| App 在后台被杀 | 恢复后无法回溯已错过的验证码（剪贴板非持久化） |

## 10. 总结

本方案充分利用现有剪贴板同步基础设施，通过增加验证码智能识别和 Mac 通知系统，实现验证码从手机到 Mac 的自动流转。

- **无需引入新 SDK 或权限**
- **无需修改中继服务器**
- **完全向后兼容**现有剪贴板同步功能
- **平台验证**: HMS Core ReadSmsManager 无法用于 HarmonyOS NEXT
- **实施成本**: 约 3 天
