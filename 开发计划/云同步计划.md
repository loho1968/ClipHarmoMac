# 云同步计划 —— 跨 WiFi/广域网剪贴板同步

> **状态**: 规划中  
> **优先级**: P2（安全与扩展）  
> **前置依赖**: 腾讯云轻量应用服务器（Node.js 镜像，已就绪）  
> **使用范围**: 个人 + 家人，非公开服务

---

## 1. 目标

在现有局域网同步基础上，新增**跨 WiFi/广域网**剪贴板同步能力：

- Mac 在家连 WiFi-A，手机在外面用 4G/5G 或 WiFi-B → 剪贴板仍然实时同步
- 两端通过**中继服务器**转发消息，无需公网 IP 或 NAT 穿透
- 局域网环境优先直连，广域网环境自动走中继

---

## 2. 当前架构 vs 目标架构

### 2.1 当前：纯局域网

```
┌──────────┐   UDP 广播发现 (19876)    ┌──────────────┐
│  Mac     │ ◄────────────────────────► │  鸿蒙手机     │
│ TCP 服务端│ ◄──── TCP 直连 (19877) ──── │  TCP 客户端   │
│ (19877)  │                           │              │
└──────────┘                           └──────────────┘
        │                                     │
        └──────── 同一局域网 (同一网段) ─────────┘
```

- Mac = TCP Server，鸿蒙 = TCP Client
- 必须在同一局域网内，无法跨网段

### 2.2 目标：局域网 + 云中继双模

```
                     ┌─────────────────────┐
                     │   腾讯云轻量服务器     │
                     │   Node.js WebSocket  │
                     │   中继服务 (wss://)   │
                     │   端口: 8443         │
                     └──────┬──────────────┘
                            │
              WebSocket     │     WebSocket
              ┌─────────────┴─────────────┐
              │                           │
         ┌────▼────┐                ┌─────▼──────┐
         │  Mac    │                │  鸿蒙手机    │
         │ WS 客户端│                │  WS 客户端   │
         └─────────┘                └────────────┘
              │                           │
              └─── 同局域网时仍可 TCP 直连 ───┘
```

- 中继服务器部署在腾讯云
- 两端各自作为 WebSocket 客户端连接中继
- 同一局域网时自动走 TCP 直连（低延迟），跨网时走中继

## 3. 中继服务器设计

### 3.1 技术选型

| 项目 | 选择 | 理由 |
|------|------|------|
| 运行时 | Node.js 20+ | 服务器镜像已含，生态成熟 |
| WebSocket 库 | `ws` | 轻量、高性能、广泛使用 |
| 反向代理 | Nginx（服务器内置） | SSL 终结 + 静态文件托管 |
| 进程守护 | PM2 | 崩溃自动重启，日志管理 |

### 3.2 核心逻辑

```
设备 A 连接 ──► 注册到房间 "room-XXXX" ──► 等待配对设备
设备 B 连接 ──► 注册到同一房间          ──► 配对成功
A 发消息 ────► 服务端转发 ──────────────► B 收到消息
```

**房间模型**：

- 每个房间由一个 **Room Key**（配对码）标识
- 同一房间内的设备互相转发消息
- 支持 1 对 1（默认）和 1 对多（扩展）
- 设备断线自动清理

### 3.3 消息协议（WebSocket 层）

在现有 `SyncMessage` JSON 之上包裹一层中继路由信息：

```json
{
  "action": "relay",
  "roomKey": "a1b2c3d4",
  "deviceId": "Mac-xxxx",
  "payload": {
    "type": "clipboardText",
    "content": "Hello",
    "timestamp": 1719600000,
    "deviceId": "Mac-xxxx",
    "mimeType": "text/plain"
  }
}
```

**客户端 → 服务端消息**：

| action | 说明 | 携带字段 |
|--------|------|---------|
| `auth` | 认证并加入房间 | roomKey, deviceId |
| `relay` | 转发剪贴板消息 | roomKey, deviceId, payload |
| `ping` | 心跳保活 | deviceId |

**服务端 → 客户端消息**：

| action | 说明 | 携带字段 |
|--------|------|---------|
| `auth_ok` | 认证成功 | pairedDeviceId（若已有配对设备） |
| `paired` | 新设备加入房间 | pairedDeviceId |
| `relay` | 转发来自配对设备的消息 | fromDeviceId, payload |
| `peer_gone` | 配对设备断开 | fromDeviceId |
| `error` | 错误信息 | message |

### 4.3 Mac 端 WebSocket 客户端设计（Swift）

```swift
// WSClient.swift — 核心 API 设计
class WSClient {
    var onConnected: (() -> Void)?
    var onDisconnected: (() -> Void)?
    var onMessageReceived: ((SyncMessage) -> Void)?
    var onPaired: ((String) -> Void)?       // 配对设备上线

    var isConnected: Bool { get }

    func connect(url: URL, roomKey: String)
    func disconnect()
    func sendRelay(_ message: SyncMessage)
    func sendPing()
}
```

- 使用 `URLSessionWebSocketTask`（iOS/macOS 13+ 原生支持）
- 自动 Ping/Pong 心跳（30 秒间隔）
- 断线自动重连（指数退避：1s → 2s → 4s → 8s → 16s，最大 30s）

### 4.4 鸿蒙端 WebSocket 客户端设计（ArkTS）

```typescript
// WSClient.ets — 核心 API 设计
export class WSClient {
  onConnected?: () => void
  onDisconnected?: () => void
  onMessage?: (msg: SyncMessage) => void
  onPaired?: (deviceId: string) => void

  get connected(): boolean

  connect(url: string, roomKey: string): void
  disconnect(): void
  sendRelay(msg: SyncMessage): boolean
  sendPing(): void
}
```

- 使用 `@kit.NetworkKit` 的 `webSocket` 模块
- 与 TCPClient 类似的心跳和重连机制

### 5.2 Room Key 生成与展示

**Mac 端（发起配对）**：
- 本地随机生成 6 位大写字母数字码（如 `A1B2C3`）
- 持久化到 UserDefaults，下次启动复用
- 在 UI 上展示，提供"复制"按钮
- 可选：生成二维码（手机扫码自动填入）

**鸿蒙端（加入配对）**：
- 手动输入 Room Key
- 或扫描 Mac 端的二维码自动填入
- 持久化到本地存储，下次自动连接
- UI 上显示当前配对状态

### 5.3 安全策略（分阶段）

| 阶段 | 安全措施 | 说明 |
|------|---------|------|
| **V1（当前计划）** | Room Key 鉴权 | 6 位随机码，防止随意接入；服务端验证 roomKey 存在才转发 |
| **V2（后续）** | 端到端加密 | AES-256-GCM，Room Key 作为密钥种子，服务端无法解密内容 |
| **V3（后续）** | TLS + Token 认证 | 生产环境强制 wss://，Token 替换简单 roomKey |

V1 阶段安全评估：
- Room Key 为 6 位字母数字 = 36^6 ≈ 21 亿种组合，暴力破解不现实
- 中继服务器仅转发已验证房间内的消息
- 服务端不存储剪贴板内容（消息即发即忘，不做持久化）

## 6. SyncManager 双模切换逻辑

```
SyncManager.start()
    │
    ├─► 启动 UDP 广播发现（局域网）
    ├─► 启动 TCP 服务端（局域网）
    │
    └─► 如果有已保存的 Room Key
        │
        ├─► 启动 WSClient，连接中继服务器
        │
        └─► 进入"双模监听"状态
                │
                ├─ 局域网发现设备 → 优先 TCP 直连
                │    └─ TCP 断开 → 自动切回 WS 中继
                │
                ├─ WS 中继配对成功 → 若 TCP 未连接则走中继
                │    └─ TCP 连接建立 → 优先切回 TCP
                │
                └─ 两者都不可用 → DISCOVERING 状态
```

**关键规则**：
1. **局域网优先**：TCP 直连延迟低、不耗服务器流量
2. **中继做后备**：TCP 不可用时自动走中继
3. **无缝切换**：切换连接方式时不丢失剪贴板消息
4. **去重兼容**：现有 `lastSentTimestamp` 去重机制在两种模式下都生效

## 9. 风险与应对

| 风险 | 可能性 | 影响 | 应对措施 |
|------|-------|------|---------|
| 服务器网络延迟高 | 中 | 体验下降 | 优先局域网直连；中继仅做备选 |
| 鸿蒙端 WebSocket API 限制 | 低 | 开发受阻 | 已确认 `@kit.NetworkKit` 支持 WebSocket（API 23+） |
| Room Key 泄露 | 低 | 第三方接入 | V2 加入端到端加密；Room Key 仅作配对标识 |
| 服务器被 DDoS/滥用 | 低 | 服务中断 | Nginx 限流；PM2 自动重启 |
| PM2 进程意外退出 | 低 | 服务中断 | PM2 自带守护 + 开机自启 |

## 11. 总结

本计划从现有 ClipboardSync 局域网同步架构出发，设计了一个**云中继双模方案**：

- **服务器侧**：Node.js + WebSocket 中继，部署于腾讯云轻量服务器
- **客户端侧**：Mac 和鸿蒙各新增 WS 客户端模块
- **连接策略**：局域网优先 TCP 直连，跨网自动走中继
- **安全**：Room Key 配对鉴权，后续可升级端到端加密
- **实施**：5 个阶段，预计总计约 5-8 天

实施过程中，始终保持**向后兼容**，不影响现有局域网同步功能的正常运行。
