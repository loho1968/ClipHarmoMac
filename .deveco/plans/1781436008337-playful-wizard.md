# 数据备份计划

## Goal

实现"按单条行程备份到云端"功能：用户在行程详情页点击"备份到云端"，将该行程及其关联的所有数据（segments、items、pictures、countries、cities、涉及的好友/货币）序列化后写入 AGC CloudDB，图片文件上传到 AGC Cloud Storage。提供"从云端恢复"功能：查询 CloudDB 获取备份列表，选择后恢复数据到本地 ORM 数据库，并根据 path 字段从 Cloud Storage 下载图片到本地沙箱。

## Scope / Non-goals

**包含：**
- AGC CloudDB + Cloud Storage SDK 集成
- 华为账号登录认证
- 单条行程数据备份（含关联表数据 + 图片文件）
- 从云端恢复单条行程（跳过本地已存在的记录）
- 备份列表查询与展示
- 备份/恢复进度 UI
- module.json5 权限配置
- 字符串资源添加

**不包含：**
- 全量备份/恢复（后续扩展）
- 自动定时备份
- 冲突解决（覆盖策略，后续扩展）
- 多用户协作/共享备份
- 离线缓存与自动同步
- 删除云端备份（v2 扩展）

## Current State And Constraints

### 现有数据模型（7个ORM表 + 关联关系）

```
TripModel (trips)
  |--< SegmentModel (segments)         segment.tripId -> trip.id
  |      |--< SegmentItemModel (items)  item.segmentId -> segment.id
  |                                     item.tripId -> trip.id
  |                                     item.extension = JSON字符串（含10种子类型）
  |--< TripPictureModel (trip_pictures) picture.tripId -> trip.id
  |                                      picture.url = 原始远程URL
  |                                      picture.path = 本地文件URI
  |                                      picture.photoId = 相册文件ID
  |--< TripCountry (trip_country)       composite PK: tripId + zhName
  |--< TripCityInfo (trip_city_info)    composite PK: tripId + siteId
  |--  TripModel.members -> FriendModel.id (逗号分隔)
```

- ORM 库：`@ibestservices/ibest-orm`（本地 HAR）
- 所有主键为 UUID 字符串，非自增（兼容 CloudDB）
- SegmentItemModel.extension 字段为 JSON 字符串，需原样备份
- TripPictureModel.path 存储的是 fileUri 格式（`file://bundleName/path`）
- TripPictureModel.url 存储原始远程 URL（可用于无图片文件时回退下载）

### 现有基础设施
- HTTP 下载：`ImageDownloadUtil`（仅用于 URL 下载图片到沙箱）
- 文件 I/O：`fs.openSync/writeSync/closeSync`（仅 ImageDownloadUtil 中使用）
- 序列化：各 ViewModel 有 `toPlainObject()` / `static fromJSON()` 方法
- 无任何云端服务集成
- 状态管理 V2：`@ComponentV2`、`@ObservedV2`、`@Trace`、`@Param`、`@Event`
- 导航：`Navigation` + `NavPathStack`

### ArkTS 约束
- 禁止 `any`、`unknown`、`as` 断言
- 禁止对象字面量展开运算符 `...obj`
- 禁止匿名内联对象字面量，必须先声明具名变量
- ORM insert 必须用 `new Model()` 构造实例
- `@Builder` 函数中不能有 const/let 声明或方法调用

### 2. CloudDB 数据模型设计

在 AGC 控制台创建 1 个 CloudDB 对象类型（NoSQL 文档型），将一条行程的所有关联数据打包为一份备份文档：

**对象类型：`TripBackup`**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | String | 主键，UUID |
| tripId | String | 原始行程ID |
| userId | String | 华为账号UID |
| trip | String | TripModel JSON字符串 |
| segments | String | SegmentModel[] JSON字符串 |
| items | String | SegmentItemModel[] JSON字符串 |
| pictures | String | TripPictureModel[] JSON字符串（path已替换为Cloud Storage引用） |
| countries | String | TripCountry[] JSON字符串 |
| cities | String | TripCityInfo[] JSON字符串 |
| friends | String | FriendModel[] JSON字符串（仅行程相关好友） |
| pictureCloudPaths | String | string[] JSON字符串，Cloud Storage上的图片路径列表 |
| backupTime | Integer | 备份时间戳（毫秒） |
| tripTitle | String | 行程标题（用于列表展示） |
| tripStartDate | String | 行程开始日期（用于列表排序） |

**设计理由：**
- 将关联数据打包为 JSON 字符串存储，避免在 CloudDB 中创建 7+ 个对象类型
- CloudDB 单文档上限 2MB，一条行程的纯数据通常远小于此限制
- 图片单独上传到 Cloud Storage，pictures 中的 path 替换为 Cloud Storage 引用路径
- tripId 作为查询索引，支持按行程恢复

**AGC 权限规则：**
```json
{
  "tripBackup": {
    "match": "userId == auth.uid",
    "allow": { "read": true, "write": true }
  }
}
```

### 3. 备份序列化层 — BackupSerializer

**文件**: `entry/src/main/ets/service/BackupSerializer.ets`

职责：将本地 ORM 数据序列化为 CloudDB 文档格式，以及从 CloudDB 文档反序列化为本地 ORM 模型实例。

**关键接口：**
```typescript
// 备份数据包
interface TripBackupData {
  id: string;              // 备份记录ID（新UUID）
  tripId: string;          // 原始行程ID
  userId: string;          // 华为账号UID
  trip: string;            // TripModel JSON
  segments: string;        // SegmentModel[] JSON
  items: string;           // SegmentItemModel[] JSON
  pictures: string;        // TripPictureModel[] JSON
  countries: string;       // TripCountry[] JSON
  cities: string;          // TripCityInfo[] JSON
  friends: string;         // FriendModel[] JSON
  pictureCloudPaths: string; // string[] JSON，Cloud Storage路径
  backupTime: number;      // 时间戳
  tripTitle: string;       // 行程标题
  tripStartDate: string;   // 开始日期
}

// 序列化：从本地ORM加载行程关联数据，生成TripBackupData
static serializeTrip(tripId: string): TripBackupData;

// 反序列化：从TripBackupData解析出各Model数组
static deserializeTrip(data: TripBackupData): DeserializedTripData;

// 图片path转换：本地path -> Cloud Storage引用路径
static convertPathToCloudRef(localPath: string, tripId: string): string;

// 图片path转换：Cloud Storage引用路径 -> 本地沙箱path
static convertCloudRefToPath(cloudRef: string): string;
```

**序列化流程：**
1. 通过 Repository 层查询 tripId 关联的所有数据
2. TripModel -> JSON.stringify (直接用 Model 字段，非 ViewModel)
3. SegmentModel[] -> JSON.stringify
4. SegmentItemModel[] -> JSON.stringify (extension 字段原样保留)
5. TripPictureModel[] -> JSON.stringify，但将其中的 `path` 字段替换为 Cloud Storage 引用路径
6. TripCountry[] -> JSON.stringify
7. TripCityInfo[] -> JSON.stringify
8. FriendModel[] -> 从 TripModel.members 解析出 Friend ID，查询后 JSON.stringify
9. 收集所有有本地 path 的图片，生成 pictureCloudPaths 列表

**反序列化流程：**
1. JSON.parse 各字段为对应类型的普通对象数组
2. 逐条创建 `new Model()` 实例，手动赋值各字段（遵守 ORM insert 规范）
3. 对于 TripPictureModel，将 Cloud Storage 引用路径暂存到 `url` 字段，`path` 留空（恢复时下载图片后填充）

```typescript
class CloudStorageService {
  // 初始化Cloud Storage
  static initialize(context: Context): void;

  // 上传单张图片到Cloud Storage
  // cloudPath格式: trips/{tripId}/pictures/{fileName}
  static async uploadImage(localPath: string, cloudPath: string): Promise<boolean>;

  // 批量上传图片（带进度回调）
  static async uploadImages(
    items: ImageUploadItem[],
    onProgress?: (completed: number, total: number) => void
  ): Promise<boolean>;

  // 下载单张图片从Cloud Storage到本地沙箱
  // 保存到 {context.filesDir}/trip_pictures/{fileName}
  // 返回本地fileUri路径
  static async downloadImage(
    cloudPath: string,
    context: Context
  ): Promise<string>;

  // 批量下载图片（带进度回调）
  static async downloadImages(
    cloudPaths: string[],
    context: Context,
    onProgress?: (completed: number, total: number) => void
  ): Promise<Map<string, string>>;  // cloudPath -> localPath映射
}
```

**图片在 Cloud Storage 中的路径约定：**
```
trips/{tripId}/pictures/{originalFileName}
```

**图片上传流程：**
1. 从 TripPictureModel.path 获取本地 fileUri
2. 将 fileUri 转换为绝对路径（`fileUri.getUriFromPath` 的逆操作）
3. 上传到 Cloud Storage 的 `trips/{tripId}/pictures/{fileName}` 路径
4. 在序列化的 pictures JSON 中，将 path 字段替换为 cloudPath

**图片下载流程：**
1. 从恢复数据的 pictures JSON 中获取 cloudPath
2. 调用 Cloud Storage SDK 下载到沙箱 `trip_pictures/` 目录
3. 获取本地 fileUri，更新 TripPictureModel.path
4. 保存到 ORM

### 5. 业务逻辑层 — BackupViewModel

**文件**: `entry/src/main/ets/viewmodel/BackupViewModel.ets`

```typescript
@ObservedV2
class BackupViewModel {
  @Trace isBackingUp: boolean = false;
  @Trace isRestoring: boolean = false;
  @Trace backupProgress: number = 0;      // 0-100
  @Trace restoreProgress: number = 0;     // 0-100
  @Trace backupList: BackupSummary[] = [];
  @Trace isLoggedIn: boolean = false;
  @Trace statusMessage: string = '';

  // 执行备份
  async backupTrip(tripId: string): Promise<boolean>;

  // 加载备份列表
  async loadBackupList(): Promise<void>;

  // 执行恢复（跳过已存在的记录）
  async restoreTrip(backupId: string): Promise<boolean>;

  // 检查登录状态，未登录则触发登录
  async ensureLoggedIn(): Promise<boolean>;
}
```

**备份流程 (`backupTrip`)：**
```
1. ensureLoggedIn() -> 未登录则弹出华为账号登录
2. BackupSerializer.serializeTrip(tripId)
   - 从Repository加载行程及关联数据
   - 序列化各Model为JSON字符串
   - 收集需上传的图片列表
3. CloudStorageService.uploadImages(pictures, onProgress)
   - 更新 backupProgress (0-50%)
   - 上传完成后，替换path为cloudPath
4. CloudDbService.createBackup(data)
   - 更新 backupProgress (50-100%)
5. 完成，statusMessage = "备份成功"
```

**恢复流程 (`restoreTrip`)：**
```
1. ensureLoggedIn()
2. CloudDbService.queryBackupDetail(backupId)
3. BackupSerializer.deserializeTrip(data)
   - 解析JSON为Model实例数组
4. 检查本地是否已存在同ID记录，跳过已存在的
5. 写入ORM
   - FriendModel: 跳过已存在的，新的insert
   - CurrencyModel: 跳过已存在的（通常已有默认货币）
   - TripModel: 跳过已存在的，新的insert
   - SegmentModel[]: insert
   - SegmentItemModel[]: insert
   - TripCountry[]: insert
   - TripCityInfo[]: insert
   - TripPictureModel[]: insert（path暂为空，待图片下载后更新）
6. CloudStorageService.downloadImages(cloudPaths, onProgress)
   - 更新 restoreProgress (50-90%)
   - 下载图片到沙箱
7. 更新TripPictureModel.path为本地fileUri，ORM save
8. 完成，statusMessage = "恢复成功"
   - 更新 restoreProgress = 100
```

### 9. 恢复时图片下载的关键逻辑

恢复流程中图片处理是最关键的环节：

```
TripPictureModel 原始记录:
  path = "file://com.loho.myapplication/data/storage/el2/base/files/trip_pictures/xxx.jpg"
  url = "https://example.com/photo.jpg"

备份时:
  1. 读取本地文件 path -> 转换为绝对路径
  2. 上传到 Cloud Storage: trips/{tripId}/pictures/xxx.jpg
  3. 在备份JSON中:
     path = "cloud://trips/{tripId}/pictures/xxx.jpg"  // Cloud Storage引用
     url = "https://example.com/photo.jpg"              // 原始URL保留

恢复时:
  1. 解析 pictures JSON，获取 cloudPath
  2. 从 Cloud Storage 下载到沙箱:
     本地路径 = {context.filesDir}/trip_pictures/{newUUID}.jpg
  3. 转换为 fileUri: fileUri.getUriFromPath(localPath)
  4. 更新 TripPictureModel.path = 本地fileUri
  5. ORM save 更新记录
```

**边界情况处理：**
- 图片文件不存在（本地path指向的文件已删除）：跳过该图片上传，仅备份 url 字段
- Cloud Storage 下载失败：记录日志，path 置空，保留 url 字段供后续手动重试
- 图片 URL 失效（url 字段的原始链接已不可访问）：path 置空，标记为缺失
- 大量图片：批量上传/下载，进度 UI 分步更新

### 10. 冲突处理（跳过已存在记录）

```typescript
// 恢复时对每条记录检查
static shouldSkipRecord(modelId: string, modelType: string): boolean {
  // 查询本地ORM是否已存在该ID的记录
  // TripModel: tripRepository.queryById(modelId)

  return exists;
```

**跳过层级：**
- TripModel: 若已存在则整条行程跳过（包括其关联数据）
- FriendModel: 若已存在则跳过（好友是全局共享的）
- CurrencyModel: 若已存在则跳过（货币是全局共享的）

## Risks And Compatibility

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| AGC CloudDB SDK (v1.0.6) 较新，API 可能不稳定 | 运行时崩溃或行为异常 | 充分测试，关注官方更新；封装服务层隔离 SDK 细节 |
| CloudDB 单文档 2MB 限制 | 含大量 items 的行程可能超限 | 监控序列化后大小，超限时提示用户拆分或压缩 |
| Cloud Storage 上传/下载速率 | 大量图片时备份/恢复耗时长 | 批量并发上传（SDK 支持），进度 UI 反馈 |
| 华为账号登录依赖系统组件 | 模拟器可能不支持 | 真机测试，模拟器上优雅降级 |
| AGC 服务仅限中国区域 | 海外用户无法使用 | 后续可扩展 REST API 方案作为海外备选 |
| `agconnect-services.json` 配置错误 | SDK 初始化失败 | 提供配置检查工具/文档 |
| ORM 与 CloudDB 数据类型映射 | 序列化/反序列化类型丢失 | 使用 JSON 字符串存储，避免直接类型映射 |
| 图片 path 格式兼容性 | fileUri 格式在不同设备可能不同 | 统一使用 `fileUri.getUriFromPath` 和反转换 |

## Rollback

1. **代码回滚**：所有新增文件在独立目录 `service/` 和 `view/` 中，删除即可；修改文件用 git revert
2. **依赖回滚**：从 oh-package.json5 移除 5 个 AGC SDK 依赖，执行 `ohpm install`
3. **配置回滚**：module.json5 移除新增权限；app.json5 移除新增配置
4. **资源回滚**：删除 string.json/color.json 中新增的条目
5. **AGC 项目**：在 AGC 控制台删除应用/项目（可选）

**回滚后影响**：回滚后备份/恢复功能不可用，但现有本地数据完全不受影响。所有 ORM 数据、图片文件、偏好设置均保持不变。
