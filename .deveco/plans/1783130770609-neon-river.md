## 目标

其他 App（相册、文件管理器、微信等）选择图片/文件 → 「用其他应用打开」或「分享」→ 选择 ClipboardSync → 手机端自动接收 → **发送到 Mac 端**。

## 范围 / 非目标

- ✅ 接收其他 App 分享的图片 → 发送 `CLIPBOARD_IMAGE` 到 Mac
- ✅ 接收其他 App 分享的文件 → 发送 `CLIPBOARD_FILE` 到 Mac
- ✅ 冷启动和热启动均支持
- ✅ 单文件和多文件均支持
- ❌ 不申请 `READ_IMAGEVIDEO` 权限（依赖分享 App 授予的临时 URI 权限）
- ❌ 不处理纯文本分享

## 权限说明

**不需要新增权限。** HarmonOS 分享机制中，发起分享的 App 在构造 `ohos.want.action.sendData` 的 Want 时，系统自动对 URI 授予接收方的临时读权限。接收方通过 `fileIo.openSync(uri)` 直接读取即可。现有权限已满足需求：`INTERNET`、`GET_WIFI_INFO`、`KEEP_BACKGROUND_RUNNING`。

## 设计

### 完整流程

```
其他 App 分享图片/文件
  → 系统生成 Want (action=ohos.want.action.sendData, uri=file://...)
  → 启动 ClipboardSync EntryAbility
  → if 冷启动:
       onCreate(want) → handleIncomingWant(want)
    if 热启动（App 已在后台）:
       onNewWant(want) → handleIncomingWant(want)
  → SyncManager.handleSharedContent(want)
     → 提取 want.uri（单文件）或 want.parameters['ability.params.stream']（多文件）
     → 对每个 URI:
        ├─ 图片: fileIo.readSync → image.createImageSource → JPEG 编码 → sendClipboardImage()
        └─ 文件: fileIo.readSync → base64 → sendFile()
  → 发送成功，记录到 history
```

### 关键文件变更

#### 1. `[module.json5](entry/src/main/module.json5)`

在 `skills` 数组追加分享 Skill（无需新增权限）：

```json5
{

  "entities": ["entity.system.default"],
  "uris": [
    { "scheme": "file", "type": "image/*" },
    { "scheme": "file", "type": "*/*" },
    { "scheme": "content", "type": "*/*" }
  ]
}
```

#### 2. `[EntryAbility.ets](entry/src/main/ets/entryability/EntryAbility.ets)`

- **新增 `onNewWant(want, launchParam)`** — 热启动接收分享
- **修改 `onCreate(want, launchParam)`** — 冷启动末尾追加 `handleIncomingWant(want)`
- **新增 `handleIncomingWant(want)`** — 过滤 `sendData` 动作 → 调用 `getSharedSyncManager().handleSharedContent(want)`

#### 3. `[SyncManager.ets](entry/src/main/ets/model/SyncManager.ets)`

- **新增 `handleSharedContent(want: Want)`** — 提取 URI 列表，确保连接已启动，逐个处理
- **新增 `processSharedUri(uri: string)`** — 读取文件，判断图片/文件类型，调用现有发送方法

### 冷启动时序处理

`onCreate` 触发时页面尚未加载（`SyncManager.start()` 在 `aboutToAppear()` 中才调用）。`handleSharedContent` 需要：

1. 检查 `this._status === SyncStatus.DISCONNECTED` → 调用 `this.start()`
2. 等待连接建立（TCP 或 WS ready）后再发送
3. 如果连接超时（例如 10 秒），放弃发送并记录日志

### ArkTS 注意事项

- `onNewWant` 是**同步**方法，内部调用异步逻辑需通过 Promise 链处理
- `want.parameters` 类型为 `Record<string, Object>`，提取 `stream` 需类型转换
- 分享 URI 权限可能在某些设备上未正确授予 → `fileIo.openSync` 异常需 catch

## 验证

| 检查项 | 方法 |
|--------|------|
| 编译通过 | `arkts_check` + `build_project` |
| 冷启动：相册分享图片 | 相册 → 选图 → 分享 → 选 ClipboardSync → Mac 收到图片 |
| 冷启动：文件管理器分享 | 文件管理 → 选文件 → 分享 → ClipboardSync → Mac 收到文件 |
| 热启动：App 后台时分享 | App 在后台 → 相册分享 → 切回 App → Mac 已收到 |

| 无权限场景 | 分享 → 系统自动授权 URI → 正常读取（不弹权限对话框） |

## 风险与兼容性

| 风险 | 说明 | 缓解 |
|------|------|------|
| 冷启动时序 | `onCreate` 时 SyncManager 未 start | `handleSharedContent` 中调用 `start()` + 等待连接 |
| URI 权限过期 | 临时授权可能有时效 | 收到后立刻读取，不延迟处理 |
| 大文件传输 | 分享的文件可能很大 | 复用现有 `sendInChunks()` 分片机制 |
