# 短信验证码同步到 Mac 实现计划

## Goal

鸿蒙手机收到验证码短信后，自动提取验证码并通过已有的剪贴板同步通道（TCP 局域网 / WebSocket 云中继）发送到 Mac。Mac 端弹出系统通知显示验证码，同时将验证码自动复制到剪贴板，用户可在 Mac 上直接 `Cmd+V` 粘贴。

## Scope / Non-goals

**Scope：**
- 新增 `verificationCode` 消息类型，复用现有 SyncManager 通信通道（TCP + WebSocket Relay）
- 鸿蒙端通过 `NotificationSubscriberExtensionAbility` 监听短信通知，正则提取验证码
- 鸿蒙端同步将验证码写入本机剪贴板（本地也可粘贴）
- Mac 端通过 `UserNotifications` 框架弹出横幅通知，自动复制验证码到剪贴板
- **降级场景**：即使手机端无法自动提取（权限被拒/API 不可用），用户手动复制验证码后 Mac 端也能识别并弹出通知
- 中继服务器无需任何改动（透明转发 payload）

**Non-goals：**
- 不实现短信发送功能
- 不支持图片验证码识别
- 不处理非验证码类短信（如银行余额提醒）
- 不改变现有剪贴板同步的核心逻辑

## Current State And Constraints

### 现有架构
项目已有完整的 **剪贴板双向同步** 基础设施：

```
鸿蒙手机 ←→ Mac
   ├── 局域网模式：UDP 发现 + TCP :19877 传输
   └── 云中继模式：WebSocket → relay-server (Node.js) → Mac
```

消息协议层：
- `SyncMessage { type, content, timestamp, deviceId, mimeType?, networkSSID?, roomKey?, relayHost? }`
- 现有消息类型：`clipboardText`, `clipboardImage`, `ping`, `pong`, `roomKeyInfo`
- 中继层 `RelayMessage { action, payload, ... }` 透明转发

### 关键平台约束

**HarmonyOS NEXT SMS API 限制：**
- `@ohos.telephony.sms` 模块**仅提供** `createMessage`, `sendShortMessage`, `getDefaultSmsSlotId`, `hasSmsCapability` —— **没有 SMS 接收/监听 API**
- HMS Core `ReadSmsManager` 是 **Android 专属** API，不适用于 HarmonyOS NEXT
- 因此采用 **通知订阅** 作为替代方案：`NotificationSubscriberExtensionAbility`（API 22+）可监听系统通知，从中提取短信验证码内容

**ArkTS 严格模式约束：**
- 禁止 `any`/`unknown`，禁止 `as` 类型断言
- 禁止 `obj[key]` 动态属性访问
- 对象字面量必须有显式类型（接口或类）
- 禁止模板字符串，使用 `+` 拼接

**权限要求（HarmonyOS）：**
| 权限 | 级别 | 用途 |
|------|------|------|
| `ohos.permission.SUBSCRIBE_NOTIFICATION` | system_grant | 订阅系统通知，读取短信内容 |

**权限要求（Mac）：**
- `UserNotifications` 框架需要用户授权才能弹出通知

**SDK 版本风险：**
- `NotificationSubscriberExtensionAbility` 需要 API 22+，项目当前 `compatibleSdkVersion` 为 21（6.0.1），可能需要提升至 22 或采用条件编译

## Design

### 1. 协议扩展

在 `SyncMessage` 中新增两个字段，两端同步修改：

**MessageType 新增：**
```
VERIFICATION_CODE = 'verificationCode'
```

**SyncMessage 新增可选字段：**
```
smsSender?: string   // 短信发送者号码（如 "106900901234"）
```

`content` 字段携带提取后的纯验证码（如 `"123456"`），`smsSender` 携带发送者号码供 Mac 端通知展示。

### 2. 端到端数据流

#### 主流程（自动提取模式）

```
[SMS到达手机]
    │
    ▼
[系统通知栏弹出] ← SMS 应用发出通知
    │
    ▼
[NotificationSubscriberExtensionAbility.onReceiveMessage()]
    │ 提取 notificationInfo.text / title
    │ 正则匹配验证码: /\d{4,8}/
    │ 找到验证码 → 回调 SyncManager
    ▼
[SyncManager.handleSmsCode(code, sender)]
    ├─ 写入本机剪贴板 (systemPasteboard.setDataSync)
    ├─ 构建 SyncMessage { type: verificationCode, content: code, smsSender: sender }
    └─ 通过现有通道发送:
        ├─ TCP 已连接 → tcpClient.send(msg)
        └─ WS 已连接 → wsClient.sendRelay(msg)
            │
            ▼
[Mac 接收] ── TCP Server / WS Client
    │
    ▼
[SyncManager.handleRemoteMessage()]
    │ type === verificationCode → 直接调用 VerificationCodeHandler
    ▼
[VerificationCodeHandler.handle(code, sender)]
    ├─ NSPasteboard.general.setString(code)  // 自动复制
    ├─ UNUserNotificationCenter 弹出通知
    │   标题: "验证码: 123456"
    │   正文: "来自: 106900901234，已复制到剪贴板"
    └─ syncHistory.addRecord(...)  // 追加到同步历史
```

#### 降级流程（手动复制模式 — 无需手机端任何 API）

```
[用户手动复制验证码到手机剪贴板]
    │
    ▼
[现有 SyncManager.checkClipboard() 轮询检测到变化]
    │ type === clipboardText
    │ content === "123456"（用户复制的验证码文本）
    └─ 通过现有通道发送到 Mac
            │
            ▼
[Mac 接收] ── SyncManager.handleRemoteMessage()
    │ type === clipboardText
    ├─ 正常写入剪贴板（现有逻辑）
    └─ ★新增★ 验证码模式检测:
        │ ClipboardMonitor 或 SyncManager 检查 content 是否匹配验证码模式
        │ 如果匹配 → 也调用 VerificationCodeHandler.handle()
        ▼
[VerificationCodeHandler.handle(code, nil)]  // sender 为空
    ├─ 剪贴板已包含验证码（前面已写入）
    └─ UNUserNotificationCenter 弹出通知
        标题: "验证码: 123456"
        正文: "已从手机同步，可直接粘贴"
```

**关键设计决策**：降级场景下，Mac 端对 **所有** `clipboardText` 消息做验证码模式检测，匹配后触发通知。这样即使手机端没有任何 SMS API，用户手动复制验证码也能得到一样的 Mac 通知体验。

### 3. HarmonyOS 端设计

#### 3a. NotificationSubscriberExtensionAbility（新建）

文件名：`entry/src/main/ets/sms/SmsNotificationSubscriber.ets`

- 继承 `NotificationSubscriberExtensionAbility`
- 实现 `onReceiveMessage(notificationInfo: NotificationInfo)`：
  1. 从 `notificationInfo` 中提取 `title` 和 `text`
  2. 用正则匹配验证码（见第 5 节）
  3. 过滤明显不是验证码的通知（如仅含中文字符无数字、超长文本等）
  4. 通过回调通知 `SyncManager` 单例

#### 3b. SyncManager 集成

在 `SyncManager.ets` 中新增：
- `smsCodeCallback?: (code: string, sender: string) => void` 属性，由 `SmsNotificationSubscriber` 设置
- `handleSmsVerificationCode(code: string, sender: string)` 方法：
  1. 构建 `SyncMessage`，`type = MessageType.VERIFICATION_CODE`，`content = code`
  2. 调用 `writeClipboardText(code)` 写入本机剪贴板
  3. 调用已有的发送逻辑发送到 Mac
  4. `addRecord(code + ' (验证码)', 'sent')` 记录到发送历史

#### 3c. 配置变更

**module.json5 新增：**
- `requestPermissions` 数组添加 `ohos.permission.SUBSCRIBE_NOTIFICATION`
- `extensionAbilities` 数组添加 `NotificationSubscriberExtensionAbility` 注册（type: `notificationSubscriber`）

**可选 SDK 升级：**
- 如需要 API 22 特性，将 `build-profile.json5` 中 `compatibleSdkVersion` 从 `"6.0.1(21)"` 提升至 `"6.0.1(22)"` 或更高

#### 3d. UI 可选增强

在 `Index.ets` 的 StatusCard 区域：
- 添加"通知权限"引导按钮，调用 `notificationExtensionSubscription.openSubscriptionSettings(context)` 打开授权页面
- 同步历史中验证码记录使用特殊样式标识（如钥匙图标 + 蓝色标记）

### 4. Mac 端设计

#### 4a. VerificationCodeHandler（新建）

文件名：`ClipboardSync/mac/ClipboardSync/VerificationCodeHandler.swift`

```swift
class VerificationCodeHandler {
    /// 提取验证码的正则模式（与手机端一致）
    private static let codePatterns: [(String, NSRegularExpression)] = {
        let patterns = [
            "验证码[：:]\\s*(\\d{4,8})",
            "验证码是\\s*(\\d{4,8})",
            "code[：:]\\s*(\\d{4,8})",
            "(\\d{4,8})\\s*（验证码）",
            "(\\d{4,8})"
        ]
        return patterns.compactMap { p in
            guard let regex = try? NSRegularExpression(pattern: p, options: []) else { return nil }
            return (p, regex)
        }
    }()
    
    /// 判断文本是否包含验证码，返回提取到的验证码
    static func extractCode(from text: String) -> String? {
        for (_, regex) in codePatterns {
            let range = NSRange(text.startIndex..., in: text)
            if let match = regex.firstMatch(in: text, options: [], range: range) {
                if match.numberOfRanges > 1,
                   let codeRange = Range(match.range(at: 1), in: text) {
                    return String(text[codeRange])
                }
            }
        }
        return nil
    }
    
    /// 处理验证码：复制剪贴板 + 弹出通知
    static func handle(code: String, sender: String?) {
        // 1. 复制到剪贴板
        let pasteboard = NSPasteboard.general
        pasteboard.clearContents()
        pasteboard.setString(code, forType: .string)
        
        // 2. 弹出系统通知
        let content = UNMutableNotificationContent()
        content.title = "验证码: \(code)"
        if let sender = sender {
            content.body = "来自: \(sender)，已复制到剪贴板"
        } else {
            content.body = "已从手机同步，可直接粘贴"
        }
        content.sound = .default
        
        let request = UNNotificationRequest(
            identifier: UUID().uuidString,
            content: content,
            trigger: nil
        )
        UNUserNotificationCenter.current().add(request)
    }
}
```

#### 4b. SyncManager 集成（关键：双重触发路径）

`handleRemoteMessage` 中的处理逻辑：

```
接收消息 → 匹配 type:
  ├─ .verificationCode:
  │     VerificationCodeHandler.handle(code: content, sender: smsSender)
  │     addRecord(content, received, sender: smsSender)
  │
  ├─ .clipboardText:                          // 现有逻辑 + 新增检测
  │     clipboard.writeText(content)           // 现有：写入剪贴板
  │     addRecord(content, received)           // 现有：记录历史
  │     ★新增★:
  │     if let code = VerificationCodeHandler.extractCode(from: content) {
  │         VerificationCodeHandler.handle(code: code, sender: nil)
  │         // 注意：不重复添加 history record，避免刷屏
  │     }
  │
  └─ 其他类型: 现有逻辑不变
```

**重要细节**：
- `clipboardText` 验证码检测**在写入剪贴板之后**进行，因为 `VerificationCodeHandler.handle()` 内部也会写入剪贴板，但写入相同内容无副作用
- 对于 `clipboardText` 触发的验证码通知，不使用 `smsSender`（来自剪贴板的文本没有发送者信息）
- 历史记录中已有 `clipboardText` 条目，不再重复添加

#### 4c. AppDelegate 通知权限

在 `ClipboardSyncApp.swift` 或 `AppDelegate.swift` 中：
- 调用 `UNUserNotificationCenter.current().requestAuthorization(options: [.alert, .sound])`

### 5. 验证码提取正则

HarmonyOS 端和 Mac 端使用相同的正则模式，确保两端识别一致：

```typescript
// 常见验证码短信模式（按优先级排列）
const CODE_PATTERNS: RegExp[] = [
  /验证码[：:]\s*(\d{4,8})/,      // "验证码：123456"
  /验证码是\s*(\d{4,8})/,          // "验证码是123456"
  /code[：:]\s*(\d{4,8})/i,       // "code: 123456"
  /(\d{4,8})\s*（验证码）/,        // "123456（验证码）"
  /(\d{4,8})/                      // 兜底：纯数字 4-8 位
]
```

按优先级依次匹配，取第一个捕获组。Mac 端使用 `NSRegularExpression` 实现等价逻辑。

### 6. 容错与降级

| 场景 | 行为 |
|------|------|
| SMS 通知中没有验证码（匹配失败） | 忽略，不做任何操作 |
| 两端未连接（无 TCP 且无 WS） | 验证码仅写入本机剪贴板，不报错 |
| Mac 通知权限被拒 | 仅复制到剪贴板，不弹出通知 |
| 手机端 SMS API 不可用（权限被拒/SDK 不足） | 用户手动复制验证码 → 同步到 Mac → Mac 端识别并弹出通知（完整降级体验） |
| Mac 端收到非验证码的 `clipboardText` | 正则不匹配 → 仅正常同步，不弹通知 |
| `clipboardText` 中包含短数字（非验证码） | 可能误触发（如复制了 "1000"），但概率低且通知无害 |

|------|------|----------|
| `NotificationSubscriberExtensionAbility` 需要 API 22，当前 SDK 21 | 可能编译失败或运行时不可用 | 提升 compatibleSdkVersion 至 22+，或使用条件编译 `@available` |
| 通知订阅权限 `ohos.permission.SUBSCRIBE_NOTIFICATION` 可能被用户拒绝 | 无法自动读取短信验证码 | 降级到手动复制模式（Mac 端 clipboardText 验证码检测已覆盖） |
| 不同 SMS 应用的 NotificationInfo 结构可能不同（`title`/`text` 字段位置） | 验证码提取可能失败 | 多字段尝试匹配（同时检查 title 和 text），加强正则覆盖面 |
| Mac 端 `clipboardText` 正则误匹配 | 普通数字文本被误识别为验证码，弹出多余通知 | 正则以 `验证码` 关键词优先，纯数字兜底仅匹配 4-8 位；可后续增加白名单/黑名单 |
| Mac UserNotifications 权限被拒绝 | 无法弹出通知 | 仅复制到剪贴板，不作通知 |
| `NotificationSubscriberExtensionAbility` 是高权限操作，华为应用商店可能要求额外审核 | 可能影响上架 | 作为可选功能，不影响核心剪贴板同步 |

## Rollback

如需回退此功能：

1. **HarmonyOS 侧：**
   - 从 `module.json5` 移除 `ohos.permission.SUBSCRIBE_NOTIFICATION` 和 `extensionAbilities` 注册
   - 删除 `sms/SmsNotificationSubscriber.ets`
   - `Protocol.ets` 中保留 `VERIFICATION_CODE` 枚举值不影响已有功能（仅会忽略该类型消息）
   - `SyncManager.ets` 中 `handleSmsVerificationCode` 删除或注释

2. **Mac 侧：**
   - `Protocol.swift` 中 `verificationCode` case 保留不影响（未知消息类型在运行时仅被忽略）
   - 删除 `VerificationCodeHandler.swift`
   - `SyncManager.swift` 中对应分支注释掉

3. **中继服务器：** 无需回退
